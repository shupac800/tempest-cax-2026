# Tempest CAX 2026 v1.2 Wave-Parameter Audit

## Root cause of the Hard-difficulty spiker bug, the v1.2 fix, and an exhaustive audit of the wave-setup system

**Project:** Tempest CAX 2026 ROM set (`tempest3`)<br>
**Document date:** July 15, 2026<br>
**Applies to:** v1.2 (`136002-222.r1` revised; all other chips unchanged from v1.1)

## Executive summary

An attract-mode observation on Hard difficulty — a spiker crawling at a small fraction of its normal speed on a purple level — was traced to a byte underflow in this build's own purple-level speed hook. Tempest stores enemy speeds as a single byte meaning (byte − 256), always negative; the hook computes the spiker speed as the base enemy speed minus 48, and Hard difficulty's stock 1/8 speed boost lowers the base-speed byte far enough that the subtraction wraps below zero. The result was a spiker speed of −6 to −19 (byte units) instead of roughly −290, on the even-numbered purple levels L98 through L112, on Hard only.

v1.2 clamps the computed spiker speed at −255, the fastest value a single speed byte can express. Every non-underflowing case — all of Easy and Medium, and Hard on odd purple levels — produces byte-identical results to v1.1.

The fix was then validated, and the rest of the wave-setup system audited, by executing the ROM's actual wave-setup routine (`CONTOUR`, entry `$92C5`) in a purpose-built 6502 emulator for all 112 levels on all three difficulty settings — 336 configurations. The audit found **no further overflow or underflow defects**. The same harness run against the pre-fix R1 image flags exactly the eight affected level/difficulty combinations and nothing else, which validates both the fix and the harness. Three latent, currently harmless quirks were identified and are documented in Section 5 so that future table edits do not trip over them.

## 1. Scope and method

The audit covers the code where difficulty and per-level parameters interact: the wave-parameter table lookup, the Easy/Medium/Hard DIP post-processing at `$9328`, this build's purple-level hook at `$D9FA`, the v1.1 enemy-count clamp at `$DA74`, and the `TIMES8` speed-scaling routine at `$93E0`. Enemy runtime code consumes the values these routines produce, so a clean wave-setup output validates the inputs to everything downstream.

Two independent techniques were used:

1. **Static table analysis.** All 28 per-wave parameter tables referenced by `WTABLE` (`$9604`) were extracted from the ROM and parsed with the record semantics from Atari's released source (`ALWELG.MAC`), checking that every parameter resolves for levels 96–112.
2. **Exhaustive emulation.** A 6502 emulator loads all ten program ROMs at their board addresses and executes `CONTOUR` for every level 1–112 at each difficulty, then validates the resulting RAM: enemy count in 1–64, all six speed values (invader, spiker, tanker, pulsar, enemy shot, fuseball) negative and not anomalously slow, shot cap at most 3, and Hard at least as fast as Medium, Medium at least as fast as Easy, for every parameter.

The random-lookup clause that stock Tempest applies from level 99 up was confirmed to be repointed in this build to level 113 and beyond, so all purple levels use their true level number for table lookups.

## 2. Speed encoding: the controlling fact

Tempest speed bytes are not two's-complement signed values. `TIMES8` unconditionally seeds the high byte with `$FF` before multiplying by 8, so a speed byte `A` always means the 16-bit value 8 × (A − 256), a range of −2048 to −8. A **smaller** byte is a **faster** enemy.

The consequence: any arithmetic that pushes a speed byte below zero does not produce "very fast" — it wraps to a value near 256, which decodes to a speed near zero. Every subtraction from a speed byte must be bounds-checked. This single fact is the root of the v1.2 bug and the reason the clamp is the correct fix.

## 3. The defect and the v1.2 fix

On purple levels (96–111 internal, L97–L112 displayed) the hook at `$D9FA` runs after the stock difficulty post-processor and computes:

```text
WINVIL  = WINVIL - (0/4/8/12)     ; base speed, faster with level
spiker  = WINVIL - 48             ; CLC / ADC #$D0 at $DA1C
```

The stock invader-speed table alternates two base bytes by wave parity on levels 65+ ($60 = −160 and $41 = −191). On Medium, the worst case is $41 − 12 − 48 = byte $05 — no wrap, with five counts to spare. On Hard, the stock post-processor first boosts the base speed by 1/8, taking $41 to $29; then $29 − (0..12) − 48 wraps below zero on every slower-parity (even-numbered) purple level, yielding spiker bytes of −6 to −19.

The fix replaces the six bytes at `$DA1C` with `JSR $DD21` plus three `NOP`s, jumping to an 11-byte helper in verified-free R1 space:

```text
DD21: SEC            ; A = WINVIL - 48; SEC/SBC #$30 produces the identical
      SBC #$30       ; result byte AND carry sense as the old CLC/ADC #$D0
      BCS +2         ; carry set = no underflow: store as before
      LDA #$01       ; underflow: clamp to -255, the fastest byte speed
      STA $0163
      RTS
```

Because `SEC SBC #$30` is bit-for-bit equivalent to `CLC ADC #$D0` whenever no borrow occurs, all previously correct cases are byte-identical to v1.1. The free space at `$DD21–$DD40` was verified: the code preceding it ends in an unconditional `JMP`, code resumes at `$DD41`, and no instruction in any program ROM references addresses in the replaced or claimed ranges.

## 4. Emulation results

Post-fix, all 336 level/difficulty configurations pass every check. Representative 16-bit speeds (more negative = faster):

| Level (Hard) | Invader | Spiker pre-fix | Spiker v1.2 | Medium spiker |
|---|---:|---:|---:|---:|
| L97 (odd) | −1432 | −1816 | −1816 | −1656 |
| L98 (even) | −1712 | **−48** | −2040 | −1904 |
| L107 (odd) | −1504 | −1888 | −1888 | −1728 |
| L108 (even) | −1784 | **−120** | −2040 | −1976 |
| L112 (even) | −1816 | **−152** | −2040 | −2008 |

The −2040 values show the new clamp engaging (8 × −255); it engages only on the eight level/difficulty combinations that previously wrapped. Run against the pre-fix R1, the harness reports exactly those eight problems — Hard at L98, L100, L102, L104, L106, L108, L110, L112 — and no others, confirming the harness detects the defect class it is designed to catch.

The originally reported sighting (a crawling spiker on the purple cardioid level, L108, Hard) corresponds to the pre-fix value −120 in the table above, about 6% of the intended speed.

## 5. Latent items found (no action taken)

Three quirks were identified that produce correct or acceptable behavior today but deserve documentation:

1. **Enemy count at L100–L112 is seeded by table fall-through.** The enemy-count table's last record ends at level 99. For levels 100+, the lookup scans past the table's end, through the two following tables' records, and matches the extended pulsars-fire record `[60–112] = 64`. The count is therefore 64 only because that unrelated record's value happens to be 64 — which is also exactly the spawn-array limit, and which the v1.1 Hard clamp keeps safe after the 1/8 boost. Any future edit to the tables between `TNYMMX` and `TWPULF` will silently change enemy counts on L100+ and should re-run this audit.
2. **A carry quirk makes L97–L100 base speed one count slow.** When the purple hook's level-offset term is zero, its two's-complement sequence leaves the carry set into the following `ADC`, adding 1 to the base-speed byte (about 0.6% slower than intended, all difficulties). Fixing it would change bytes on every difficulty for an imperceptible gain; left as is.
3. **Purple enemy-shot speed is pinned at −1792 on all difficulties** by the hook, slightly slower than the −2040 that odd-parity levels 65–96 compute. This reads as a deliberate design choice — it also keeps the value away from the byte boundary — and is flagged only for completeness.

Outside the wave-setup path, the difficulty DIP bits are read in exactly three other places: the rating-message display index and two DIP-change detectors. None performs arithmetic capable of overflow.

## 6. Audit boundaries

The audit exhaustively covers everything downstream of the difficulty switch: all parameters seeded at wave setup, on every level, at every setting. It does not extend to purple-level features with no difficulty interaction — the extra superzapper, colors, start-of-level bonus, and high-score display — which consume none of the audited parameters. The emulator, harness, and checks are reproducible from the ROM images alone plus the public Atari Tempest source (`ALWELG.MAC`) used for symbol names.

## Conclusion

The defect fixed by v1.2 was a byte underflow inherent in subtracting from Tempest's (byte − 256) speed encoding without a bounds check, exposed only when Hard difficulty's stock speed boost consumed the remaining headroom on the slower-parity purple levels. The clamp at `$DD21` fixes it with byte-identical behavior in every previously correct case, and the exhaustive 336-configuration emulation shows the wave-setup system now contains no overflow or underflow at any level on any difficulty setting. The remaining caution is structural rather than behavioral: the enemy-count fall-through described in Section 5 means future table edits in that region must be followed by a re-run of this audit harness.
